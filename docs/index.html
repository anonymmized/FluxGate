How it works

FluxGate acts as a local HTTPS proxy. Your app talks to FluxGate; FluxGate optimises the request and forwards it upstream.

One-line setup

# 1. Install (generates CA + config) curl -fsSL https://raw.githubusercontent.com/anonymmized/FluxGate/main/install.sh | bash # 2. Start fluxgate --config ~/.fluxgate/config.toml # 3. Point your app at the proxy export HTTPS_PROXY=http://127.0.0.1:8080 # That's it.

Everything you need to cut costs

Smart Response Cache

Identical queries return instantly from memory — no API call, no cost. LRU + TTL with optional Redis backend for multi-instance deployments.

Automatically keeps only the last N messages in the context window. Cuts input token count on long conversations by 50–80%.

Install in 30 seconds

The installer downloads the binary, generates a local CA, trusts it in your OS keychain, and writes a default config.

macOS & Linux

curl -fsSL https://raw.githubusercontent.com/anonymmized/FluxGate/main/install.sh | bash

Installs to ~/.fluxgate/. Run with sudo to trust CA system-wide.

Docker

docker run -d \ -v ~/.fluxgate:/certs:ro \ -p 8080:8080 -p 9090:9090 \ ghcr.io/anonymmized/fluxgate:latest \ --mitm \ --mitm-ca-key /certs/ca.key.pem \ --mitm-ca-cert /certs/ca.cert.pem

Frequently asked questions

Is it safe to route my API keys through FluxGate?
FluxGate runs locally — on your machine or inside your network. Your traffic never leaves your infrastructure. FluxGate never stores API keys or response content. Structured logs contain only metadata (host, method, path, latency). See SECURITY.md for the full trust model.
Does it work with streaming responses (Server-Sent Events)?
Yes — streaming responses are relayed byte-for-byte without buffering. FluxGate automatically skips caching for SSE streams (Content-Type: text/event-stream).
What languages / SDKs are supported?
Any HTTP client that supports HTTP CONNECT proxies: Python (openai, anthropic), Node.js, Go, Ruby, Java, curl, etc. See examples/ for ready-to-run code.
How much traffic can it handle?
FluxGate uses async C++20 (Asio) with a per-session strand model. A single instance handles thousands of concurrent connections on commodity hardware. Each session adds <1ms overhead for a cache hit.
Does it support multiple instances / horizontal scaling?
Yes — use Redis as the cache backend (cache.backend = "redis") so all instances share the same cache. Each instance is stateless otherwise.
